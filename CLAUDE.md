# hahaton-2026 — инструкции для агентов

Проект: **AI CEO / Founder Strategist** — агент, превращающий бизнес-идею в скоринговый
отчёт валидации (4 категории ×25 → /100 + gaps) с Multi-LLM панелью. Полный план — [PLAN.md](PLAN.md).

Команда работает **параллельно в одном репозитории**. Чтобы не рассинхронизироваться,
соблюдай обязательный цикл ниже — ВСЕГДА и АВТОМАТИЧЕСКИ, без напоминаний пользователя.

---

## ⛔ ОБЯЗАТЕЛЬНЫЙ ЦИКЛ (выполнять всегда)

### ПЕРЕД любыми манипуляциями в репозитории

1. **Синхронизируйся с удалёнкой:** `git pull --rebase origin main`.
   Делай это в начале сессии и перед каждой правкой — коллеги пушат параллельно.
2. **Открой и прочитай `plan.html` — всегда в английской версии** (`plan.html?lang=en`
   в браузере, либо словарь `I18N.en` / JSON-острова `#data-progress` и `#data-workstreams`
   в исходнике): пойми текущий статус проекта, что уже сделано, что в работе, кто чем занят.
   **Отталкивайся от этой актуальной информации**, а не от устаревших предположений.
   При расхождении плана и кода — доверяй коду, но отметь это в логе.
3. **Сразу застолби задачу claim-коммитом — ДО написания кода.** Как только в начале
   сессии стало понятно, над чем будешь работать, обнови JSON-острова в `plan.html`
   (claim, см. ниже) и **сразу запушь этот коммит**. Если на этот момент менять в
   `plan.html` нечего или хочешь объявить намерение мгновенно — делай пустой коммит:
   `git commit --allow-empty -m "🔨 start <id>: <что начинаешь>"`, затем push.
   Смысл: команда видит в `plan.html` / истории, кто и над чем работает,
   **ДО того как задача уже сделана**, а не после.
4. Только после пуша claim-коммита начинай саму работу.

### ПОСЛЕ любых манипуляций в репозитории

Порядок git-операций важен (`pull --rebase` падает при незакоммиченных изменениях),
поэтому коммить ДО rebase:

1. **Обнови `plan.html`** — отрази свой прогресс в JSON-островах (см. ниже). Это часть
   каждого изменения, а не отдельная задача.
2. `git add -A` → `git commit -m "..."`.
3. **Синхронизируйся:** `git pull --rebase origin main` (вдруг кто-то запушил, пока ты работал; конфликты разрули).
4. `git push origin main`.
5. При желании проставь `commit` в добавленной записи прогресса
   (`git rev-parse --short HEAD`) отдельной мелкой правкой.

> Цель: каждый агент всегда видит свежий статус и всегда оставляет свежий статус.
> Никто не работает вслепую и никто не теряет чужой прогресс.

## 🔒 Claim-коммит ПЕРЕД стартом сабтаски (чтобы не дублировать работу)

Как только ты **начинаешь** выполнять какой-либо процесс/пункт из плана — сразу, **до**
самой работы, застолби его отдельным маленьким коммитом, чтобы другие агенты видели,
что он в работе и кем.

Этот коммит — **claim-коммит**: он меняет только JSON-острова `plan.html` (`#data-roadmap`,
`#data-progress`, опционально `#data-workstreams`) и не содержит код/реализацию задачи.
Он не обязан быть буквально пустым: статус в `plan.html` должен измениться. Если статус уже
изменён без файлового diff, допустим `git commit --allow-empty -m "Claim <id>: <task>"`,
но нормальный путь — коммит с изменением `plan.html`.

1. Открой вкладку **Roadmap** (остров `#data-roadmap` в исходнике `plan.html`), найди нужный
   саб-таск по `id` (напр. `A3`) и выставь ему `"status": "in-progress"` и `"owner": "<твоё имя>"`.
   (Опц.) если у воркстрима в целом сменился статус — обнови и `#data-workstreams`.
2. В `#data-progress` добавь запись о старте (в начало массива), с `id` саб-таска в summary:
   `{ "date":"…", "time":"HH:MM", "author":"<имя>", "workstream":"A|B|C|D", "summary":"🔨 started A3: <что именно начал>", "commit":"—" }`.
3. **Сразу сделай отдельный claim-коммит и push — ДО основной работы:**
   `git add plan.html && git commit -m "🔨 start A3: market sizing" && git pull --rebase origin main && git push origin main`.
   После этого только приступай к коду/докам самой задачи.
   Если обновить `plan.html` пока нечем — застолби намерение пустым коммитом
   (`git commit --allow-empty -m "🔨 start <id>: …"`) и тоже сразу запушь.
4. **Не смешивай claim и реализацию.** Если в коммит попали файлы задачи, остановись и раздели изменения
   на два коммита: сначала claim/status, потом реализация.
5. **Перед стартом проверь:** если нужный саб-таск уже `in-progress` с другим `owner` —
   не дублируй; возьми другой `id` или согласуй с владельцем.
6. Закончив задачу — отдельным финальным коммитом выставь саб-таску `"review"`/`"done"` в
   `#data-roadmap`, добавь запись завершения в `#data-progress`, затем `pull --rebase` и `push`.

Полная декомпозиция проекта на саб-таски (id F1…D5) — во вкладке **Roadmap** (`#data-roadmap`).

> Правило простое: **claim-коммит → push → потом работай → completion-коммит → push.**
> Статус и owner появляются раньше кода, а завершение задачи фиксируется отдельно после реализации.

## Как обновлять `plan.html` (только JSON-острова, НЕ верстку/CSS)

1. **`#data-progress`** — добавь запись **в начало** массива. `summary` пиши **на английском**
   (страница двуязычная, лог — общий и читается агентами по-английски):
   ```json
   { "date":"YYYY-MM-DD", "time":"HH:MM", "author":"<имя>", "workstream":"A|B|C|D",
     "summary":"<one line, in English>", "commit":"<short-sha|—>" }
   ```
   Дату бери из системного контекста сессии.
2. **`#data-workstreams`** — если статус твоего воркстрима изменился, обнови `status`
   (`todo` → `in-progress` → `review` → `done`) и `owner`. Заголовки воркстримов двуязычны
   (`title_uk` / `title_en`) — их менять обычно не нужно.

JSON-острова редактируются легко и почти не дают merge-конфликтов. Контентные секции
плана статичны (зеркало PLAN.md) — их трогать не нужно.

## Командная синхронизация

- `packages/contracts` (`@hahaton/contracts`) — **замороженный контракт**. Менять только
  синком со всей командой (есть `pnpm generate:contracts`).
- Воркстримы пишут в разные пакеты, чтобы не конфликтовать. Не трогай чужие пакеты без необходимости:
  - **A** — пайплайн агента: `packages/agent`, шаги + grounding (`packages/integrations` — LLM-gateway/web_search; план порта — `specs/llm-gateway-plan.md`).
  - **B** — стрим/персист: `apps/api` (SSE `/agent/stream`, раздача статики) + `packages/store`, движок `packages/unit-economics`.
  - **C** — UI: `apps/web` (стек TBD).
- Коммиты мелкие и частые, push сразу — это снижает конфликты при параллельной работе.

## Стек

Монорепа **pnpm + turbo** (`apps/*`, `packages/*`). Node 20 + TypeScript (ESM), бэкенд — Express
в `apps/api`, `@anthropic-ai/sdk`, деплой Cloudflare (сервис `@hahaton/api`, порт в работе). ESM-импорты — с расширением `.js`.
Скрипты из корня: `pnpm dev` / `pnpm build` / `pnpm typecheck` / `pnpm generate:contracts`.
Структура и зависимости пакетов — в [README.md](README.md).
